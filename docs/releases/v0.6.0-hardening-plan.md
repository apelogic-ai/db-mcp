# v0.6 Hardening Plan

**Status:** Proposed  
**Scope:** Deterministic execution lifecycle, capability unification, and stronger agent/runtime reliability.

## Goals

- Make tool behavior deterministic across SQL, file, SQL-like API, and REST API connectors.
- Enforce protocol-first workflow without sacrificing execution reliability.
- Normalize sync/async query execution to one state model and one result/error contract.
- Keep existing tools backward-compatible while routing through a single execution core.

## Problems Observed in v0.5

- Capability defaults are duplicated in multiple places (`server.py`, `connectors/__init__.py`), creating drift risk.
- Execution surfaces are fragmented (`run_sql`, `api_execute_sql`, connector-level polling), with inconsistent result envelopes.
- Async behavior is partly hidden inside connector implementations, limiting observability and resumability.
- In-memory query state is process-bound and non-resumable across restarts.
- Transport/setup failures are hard to diagnose quickly (no deterministic preflight/doctor command).

## Target Architecture

### 1) Unified Execution Contract

All SQL-capable connectors expose one lifecycle:

`PRECHECK -> VALIDATED -> SUBMITTED -> RUNNING -> (SUCCEEDED | FAILED | CANCELLED | TIMED_OUT)`

With a single typed envelope:

- `ExecutionRequest`
- `ExecutionHandle`
- `ExecutionResult`
- `ExecutionError` (`AUTH`, `TRANSPORT`, `VALIDATION`, `ENGINE`, `TIMEOUT`, `RATE_LIMIT`, `POLICY`, `TOOLING`)

### 2) Capability Single Source of Truth

- One module for capability defaults/normalization.
- Server tool registration and runtime checks must use the same normalization path.

### 3) Explicit Async Model

- SQL-like APIs should return a job handle from submit and use explicit polling.
- Connector internals may still optimize, but db-mcp surface remains uniform.

### 4) Persistent Execution State

- Move from in-memory-only query lifecycle to persistent store (SQLite per connection).
- Support resumable polling and idempotency keys.

## Phased Delivery

## v0.6.0 (Foundation)

- Add shared capability normalization module and remove duplicated defaults.
- Add execution model package (`execution/models.py`) as the canonical contract.
- Keep existing tools and responses compatible.
- Add regression tests for capability parity and execution model validation.

**Acceptance Criteria**

- Capability defaults are defined once and consumed by both server and connector runtime logic.
- Existing tests for capability registration/runtime compatibility pass.
- Execution models are importable and validated by tests.

## v0.6.1 (Execution Core + Persistence)

- Introduce execution engine/state machine module.
- Add persistent execution store (SQLite) with lifecycle transition log.
- Route `run_sql`/`get_result` through execution engine.
- Add idempotency key support.

**Acceptance Criteria**

- Executions survive process restart and remain pollable.
- One execution id can be traced end-to-end (submit/poll/final result).
- Error responses are typed and stable.

## v0.6.2 (Connector and Transport Hardening)

- Refactor SQL-like API execution to explicit submit/status/results adapter behavior.
- Add CLI `doctor --json` preflight checks (auth, connection resolve, execute test, poll test).
- Add stdio MCP handshake E2E tests (real process spawn + initialize/tools/list/tools/call).

**Acceptance Criteria**

- Doctor provides deterministic pass/fail with machine-readable diagnostics.
- Transport handshake failures are reproducible in automated tests.
- Async polling behavior is consistent across connectors.

## v0.6.3 (Agent Determinism + Surface Optimization)

- Add server-side execution policy gate for SQL execution tools.
  - Enforce connector capability checks, write confirmation policy, and optional
    protocol acknowledgment before execution.
- Add protocol acknowledgment marker flow (`protocol` tool / shell protocol read)
  to support deterministic policy enforcement.
- Define tool-surface profiles and reduce default agent-exposed tools.
  - Keep compatibility aliases, but route agents toward a canonical safe execution path.
- Add tool-surface regression tests per profile (expected tool names by mode/capability).
- Export versioned response contracts as JSON Schema files for CI/release diffing.

**Acceptance Criteria**

- SQL execution tools reject requests deterministically when policy preconditions fail.
- When protocol acknowledgment gating is enabled, `run_sql` execution is blocked until
  protocol acknowledgment is recorded.
- Default tool profile exposes a reduced, intentional surface for query workflows.
- Tool registration drift is caught by automated profile/surface tests.
- `run_sql` / `get_result` response schemas are committed under versioned paths and
  CI fails when generated schemas drift from models.

## Compatibility Strategy

- Keep `run_sql` and `api_execute_sql` public tools in v0.6.x.
- Implement `api_execute_sql` as an alias/delegation path over unified execution internals.
- Preserve current response fields while adding typed metadata (`state`, `error.code`) progressively.

## Test Strategy

- Unit:
  - capability normalization
  - state transition validity
  - error normalization mapping
- Integration:
  - `run_sql` + `get_result` lifecycle across connector types
  - SQL-like API async adapter submit/poll terminal states
- E2E:
  - stdio MCP startup handshake
  - real tool invocation pipeline (`initialize -> tools/list -> tools/call`)
  - policy-gated execution behavior with and without protocol acknowledgment
  - tool-surface snapshot/profile tests

## Non-Goals (v0.6.x)

- Full QueryStore protocol migration (separate track).
- Major changes to knowledge vault schema.
- Removing existing tool names.
