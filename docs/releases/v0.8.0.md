# db-mcp v0.8.0

Release date: 2026-03-25

## Summary

`v0.8.0` introduces the first metric-first semantic runtime in `db-mcp`, centered on `answer_intent(...)`. This release adds typed semantic planning, approved metric/binding execution, executable rule-policy support for selected business-rule families, and shared semantic-runtime exposure across the main product surfaces while keeping one target connection per request.

## Highlights

- Added `answer_intent(...)` as the primary semantic-serving path for approved metrics and bindings.
- Added typed semantic and orchestration models so responses include:
  - `meta_query`
  - `resolved_plan`
  - `provenance`
  - `confidence`
  - `result_shape`
- Added connection-local metric bindings and executable semantic policy support for:

- Extended the same semantic engine into code/runtime and daemon-facing surfaces.
- Fixed knowledge-usage analytics so semantic runs are visible in traces and insights again.

## Why this release matters

The product now has a governed semantic-serving path that is measurably stronger than prompt-only SQL generation on metric-shaped questions. Instead of asking the model to rediscover metric semantics on every request, `db-mcp` can now:

1. resolve intent against approved connection-local metrics,
2. produce a typed semantic plan,
3. compile that plan into a reviewed connection-bound execution plan,
4. execute through the existing guarded SQL path,
5. return answer plus structured provenance and confidence.

This release does **not** attempt cross-connection routing or multi-source aggregation. It establishes the first trustworthy semantic runtime slice on top of one explicit target connection per request.

## Upgrade notes

- `answer_intent(...)` is the intended semantic-serving surface in `v0.8.0`.
- Existing prompt-driven `db_mcp` and daemon SQL-writing paths remain available, but they should be treated as compatibility/comparison surfaces rather than semantically equivalent replacements for `answer_intent`.
- This release stays single-target per request: one explicit connection is resolved and executed for each semantic answer.
- Candidate semantic importer output remains governed operator input; it does not imply auto-promotion to approved production semantics.

## Known issues

- The legacy `db_mcp` and `runtime_daemon` prompt-driven flows can still diverge from approved metric definitions on some hard cases even when `answer_intent` resolves them correctly.
- The longstanding `PytestReturnNotNoneWarning` in `tests/test_database.py::test_connection` remains unchanged.
- In some local macOS sandboxed environments, Playwright Chromium launch can fail before UI tests execute even when the app/harness is healthy. This does not affect runtime behavior or packaging of the Python/core release.
