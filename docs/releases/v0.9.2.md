# Release v0.9.2 - Vault Write Unification

## Overview

This release centralises all knowledge vault write and delete operations into a single schema registry layer (`db_mcp_knowledge/vault/schema_registry.py`). Previously, validation logic and side effects (dual-writes, cross-catalog checks, group resolution) were scattered across 8 individual MCP tool handlers. All 8 handlers now delegate through `vault_write_typed()` or `vault_delete_typed()`, making the registry the single authoritative owner of write behaviour. Public MCP tool names and signatures are unchanged.

## Highlights

- **Schema registry** — new `vault_write_typed()` / `vault_delete_typed()` functions centralise validation, pre-hooks, and post-hooks for all knowledge writes
- **Automatic dual-writes** — `query_approve` feedback log entry and `query_feedback` corrected-example save are now registered post-hooks, not inline code in the tool handler
- **Dedup guard on `query_add_rule`** — duplicate business rules are now rejected before touching disk
- **Cross-catalog validation on `metric_binding`** — pre-hook verifies referenced metric and dimensions exist before the binding is written
- **Consistent delete path** — `metrics_remove` routes through `vault_delete_typed` with a pre-hook that raises clearly on missing items rather than returning a silent error dict
- **core and mcp-server metrics.py in sync** — both packages now share the same registry-delegating implementation

## New Features

- `db_mcp_knowledge/vault/schema_registry.py` — `SchemaEntry`, `DeleteEntry`, `vault_write_typed()`, `vault_delete_typed()`, `register`/`lookup`, `register_delete`/`lookup_delete`
- 7 write entries: `approved_example`, `corrected_feedback`, `business_rule`, `metric`, `dimension`, `metric_binding`, `gap_dismissal`
- 2 delete entries: `metric_deletion`, `dimension_deletion`
- New docs: `docs/api-connector-duckdb-cleanup.md`, `docs/knowledge-pipeline.md`
